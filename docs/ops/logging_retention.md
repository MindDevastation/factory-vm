# Logging & Retention Operations

## 1. What this policy covers

This policy covers local, project-owned retention for logs and disposable artifacts generated by factory-vm runtime workflows. It defines how `scripts/ops_retention.py` scans and prunes old files and how disk pressure is evaluated before urgent retention behavior is enabled.

## 2. Log classes and where they are stored

- **Project log files**: `FACTORY_LOG_DIR` (default: `${FACTORY_STORAGE_ROOT}/logs`).
- **Preview artifacts**: `${FACTORY_STORAGE_ROOT}/previews`.
- **Export artifacts**: `exports/` and `data/exports/` under repo root.
- **Transient reports/artifacts**: `qa/` and `data/qa/` under repo root.
- **Workspace artifacts**:
  - terminal workspace files: `${FACTORY_STORAGE_ROOT}/workspace/terminal`
  - failed workspace files: `${FACTORY_STORAGE_ROOT}/workspace/failed`

## 3. Project-owned log rotation policy

`ops_retention.py` always evaluates disk status and then applies age-based retention to disposable project artifacts:

- previews older than `FACTORY_RETENTION_PREVIEW_HOURS`
- exports older than `FACTORY_RETENTION_EXPORT_DAYS`
- transient reports older than `FACTORY_RETENTION_TRANSIENT_REPORT_DAYS`
- terminal workspace files older than `FACTORY_RETENTION_TERMINAL_WORKSPACE_HOURS`
- failed workspace files older than `FACTORY_RETENTION_FAILED_WORKSPACE_DAYS`

Project log files under `FACTORY_LOG_DIR` are only added to deletion candidates in **urgent mode** and only when disk status is critical (`FAIL`).

## 4. Journald retention hardening policy

Deploy `deploy/systemd/journald-retention.conf` to enforce bounded journald retention:

- `SystemMaxUse=512M`
- `RuntimeMaxUse=128M`
- `MaxRetentionSec=30day`

## 5. Disposable vs protected artifact policy

Disposable artifacts (eligible for retention deletes):

- preview files
- export files
- transient QA/report files
- terminal/failed workspace files
- log files in `FACTORY_LOG_DIR` only during urgent+critical flow

Protected artifacts (not touched by `ops_retention.py`):

- database files (for example `FACTORY_DB_PATH`)
- backup snapshots under `FACTORY_BACKUP_DIR`
- source code and deploy config files

## 6. Retention runner commands

- `python scripts/ops_retention.py scan`
  - report-only candidate scan; no deletes
- `python scripts/ops_retention.py run`
  - apply configured retention deletes
- `python scripts/ops_retention.py run --urgent`
  - urgent mode; extra log-file pruning is effective only when disk status is critical (`FAIL`)

## 7. Disk warning/critical thresholds

Disk thresholds are sourced from:

- `FACTORY_SMOKE_DISK_WARN_PERCENT`
- `FACTORY_SMOKE_DISK_WARN_GIB`
- `FACTORY_SMOKE_DISK_FAIL_PERCENT`
- `FACTORY_SMOKE_DISK_FAIL_GIB`

Default evaluator behavior:

- `PASS` when above warning/fail thresholds
- `WARN` when below warning threshold
- `FAIL` when below fail threshold

## 8. What operations are blocked under critical disk

When app-level disk status is critical, the write-heavy guard blocks only the following operations:

- render-heavy enqueue/start
- retry paths that create new render-heavy work
- analyze batch start
- large transient export generation

The following operations remain allowed under critical disk:

- cleanup
- retention
- smoke
- backup verify/restore
- read-only UI/API operations
- safe recovery actions that are not write-heavy

## 9. Structured retention log event meanings

Structured retention and disk-pressure events use the following emitted event names:

- `retention.scan.start`: retention scan requested/started
- `retention.scan.complete`: retention scan finished with summary counters
- `retention.delete.success`: a retention delete completed successfully
- `retention.delete.failure`: a retention delete attempt failed
- `retention.skip`: retention action skipped (for example, urgent-only behavior requested when preconditions are not met)
- `retention.urgent.start`: urgent retention flow entered
- `retention.urgent.complete`: urgent retention flow completed
- `disk.warning`: disk state evaluated as warning
- `disk.critical`: disk state evaluated as critical

## 10. Safe rollout guidance

1. Run scan/report-only first:
   - `python scripts/ops_retention.py scan`
2. Verify outputs:
   - review `retention.scan.start` / `retention.scan.complete` and any `retention.skip` records
3. Enable destructive run:
   - `python scripts/ops_retention.py run`
4. Use urgent mode only for active pressure:
   - `python scripts/ops_retention.py run --urgent`
