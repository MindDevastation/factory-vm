REQ_EPIC6A_TELEGRAM_OPERATOR_LAYER_v1.0

1. Название

Epic 6A — Telegram Operator Layer

2. Цель

Добавить в factory-vm полноценный Telegram-based remote operator console, который позволяет оператору:
- получать критичные события и summaries;
- безопасно просматривать состояние фабрики;
- выполнять ограниченный, auditable и guardrailed набор operator actions;
- управлять publish / readiness / recovery / queue / ops сценариями удаленно;
- работать с фабрикой без постоянного присутствия в web UI.

3. Зачем делаем

Сейчас Telegram в проекте уже присутствует как важная operator surface для части publish workflows и уведомлений, но этого недостаточно для роли полноценной remote operator console.

Epic 6A нужен, чтобы:
- убрать зависимость от постоянного открытия web UI для критичных действий;
- сделать удаленный контроль фабрики штатным сценарием;
- сократить latency между проблемой и реакцией оператора;
- превратить Telegram из узкого notification/action adjunct в first-class operator layer;
- обеспечить безопасный remote control flow без превращения Telegram в неуправляемый или слишком “магический” интерфейс.

4. Зафиксированный архитектурный контекст

4.1. Epic 6A — отдельный эпик.
Он не растворяется в Epic 6 и не подменяет собой UX redesign.

4.2. Telegram bot в этом эпике = Remote operator console, а не просто notification bot.

4.3. Epic 6A не является “чат-агентом общего назначения”.
V1 не предполагает free-form autonomous natural-language control как основной механизм управления.
Основной режим Epic 6A v1:
- structured commands
- structured callbacks
- structured summaries
- guarded operator actions

4.4. Web UI остается primary canonical administration surface.
Telegram — это полноценная secondary operator surface для удаленной работы, а не новый единственный source of truth.

4.5. Все Telegram-triggered actions должны оставаться:
- auditable
- permission-checked
- idempotent
- bounded by existing product semantics

4.6. Epic 6A не должен silently invent новые доменные действия.
Telegram layer должен вызывать уже существующие или явно зафиксированные product actions других эпиков, а не создавать скрытую альтернативную логику.

5. Scope v1

В Epic 6A v1 входит:

5.1. Telegram operator identity and chat binding
- привязка Telegram operator identity к product operator identity;
- binding user/chat/thread contexts;
- permission-aware routing;
- trusted operator enrollment / revocation.

5.2. Telegram notifications and operator inbox
- critical alerts;
- queue events;
- publish/manual handoff events;
- readiness/problem alerts;
- digest/summaries;
- acknowledgment model.

5.3. Telegram read surfaces
- publish queue views;
- readiness/problem views;
- release/job status drilldowns;
- recovery/ops summary;
- channel/factory overview summaries;
- links back to web UI where applicable.

5.4. Telegram action surfaces
- publish-related actions;
- manual handoff actions;
- retry/ack/reschedule-like actions where allowed;
- ops/recovery controls where safe;
- batch-safe operator actions with explicit guardrails.

5.5. Audit / safety / idempotency
- action traceability;
- callback expiry;
- explicit confirmation for dangerous operations;
- dedupe / idempotency;
- permission enforcement;
- action failure visibility.

5.6. Message lifecycle / state management
- pending/active/resolved action messages;
- stale message behavior;
- follow-up messages;
- conflict handling when underlying state changed elsewhere.

6. Non-goals v1

В Epic 6A v1 не входит:

6.1. Общий redesign интерфейса, design system и animations.
Это Epic 6.

6.2. Analytics-first dashboard epic.
Analytics center остается в Epic 5A.

6.3. Полностью свободный natural-language orchestration agent.
V1 — structured operator console.

6.4. Полноценное редактирование всех сущностей продукта через Telegram.
V1 покрывает operator-relevant read surfaces и ограниченный набор safe actions.

6.5. Замена web UI как основного административного интерфейса.

6.6. Массовые destructive/admin-only actions без многоступенчатых guardrails.

6.7. Скрытое добавление новых доменных semantics вне уже принятых REQ соседних эпиков.

7. Ключевые product-решения

7.1. Telegram в v1 — полноценная remote operator surface, но не primary source of truth.

7.2. Любое mutating действие в Telegram должно:
- маппиться на канонический доменный action;
- проходить permission check;
- иметь audit trail;
- быть idempotent;
- быть защищено от stale state.

7.3. Telegram layer в v1 строится как:
- event-driven notifications
- structured command/callback interactions
- compact read views
- safe action flows
а не как general AI operator.

7.4. Для опасных/чувствительных действий обязателен confirmation step.

7.5. Если underlying state уже изменился в web UI или другим worker flow, Telegram action не должен silently overwrite state.
Он должен:
- показать stale/conflict result;
- не выполнять небезопасное действие;
- направить оператора в актуальный context.

7.6. Epic 6A должен быть useful даже без Epic 6 redesign.
Он должен оставаться функциональным и операторски ценным в текущем UI-контексте.

8. Каноническая модель operator ownership

8.1. Product operator identity — канонический owner прав.
Telegram identity — привязываемый remote surface identity.

8.2. Chat/thread binding не заменяет product operator identity, а лишь разрешает конкретный remote surface context.

8.3. Mutating authority определяется:
- operator identity
- permission scope
- action class
- target entity/state
- confirmation/idempotency constraints

9. Notification / inbox model

9.1. Telegram v1 должен поддерживать:
- immediate critical alerts
- actionable alerts
- summaries/digests
- unresolved items follow-up
- resolved/closed updates where meaningful

9.2. Уведомления должны иметь severity / category / target context.

9.3. Telegram messages не должны жить в вакууме.
Каждое actionable сообщение должно знать:
- к какому entity/context оно относится;
- какой action set разрешен;
- не стал ли message stale.

10. Read surface model

10.1. Обязательные read surface families:
- factory overview
- channel overview
- publish queue / manual handoff
- readiness / blockers
- release/job detail summary
- recovery / worker / ops summary
- problem/anomaly list

10.2. Telegram read surfaces должны быть компактными и operator-first:
- краткая суть
- причина/риск
- доступные действия
- ссылка в web UI, если нужен полный контекст

11. Action model

11.1. Telegram v1 должен поддерживать operator actions как минимум в таких классах:
- publish decision actions
- manual handoff actions
- queue/retry/ack actions
- safe recovery/ops actions
- limited batch actions

11.2. Действия делятся на:
- read-only
- low-risk mutation
- guarded mutation
- high-risk guarded mutation

11.3. Для guarded/high-risk actions обязателен confirmation flow.

12. Safety / audit / hardening model

12.1. Любой Telegram action должен логироваться как product audit event.

12.2. Любой callback/action token должен иметь:
- expiry
- idempotency behavior
- stale-state protection
- operator identity linkage

12.3. Ошибки должны быть operator-readable:
- permission denied
- state changed
- action expired
- entity not found
- already completed
- currently forbidden by policy
- underlying system unavailable

12.4. Telegram layer не должен silently degrade в unsafe behavior.
При uncertainty — fail closed, not fail open.

13. Scope boundaries with adjacent epics

13.1. Epic 6A не заменяет Epic 3.
Он лишь дает remote surface для publish-domain flows.

13.2. Epic 6A не заменяет Epic 5.
Он не становится visual-design/control epic.

13.3. Epic 6A не заменяет Epic 5A.
Analytics notifications возможны позже, но analytics center остается отдельным доменом.

13.4. Epic 6A не заменяет Epic 6.
Красивый unified control center и design system остаются отдельным эпиком.

13.5. Epic 6A не заменяет Epic 7.
Документация и runbook consolidation остаются отдельным эпиком.

14. Acceptance criteria на уровне эпика

Epic 6A считается успешно закрытым на уровне REQ/продукта, если:

14.1. Telegram bot является полноценной remote operator console, а не только notification bot.

14.2. Есть канонический operator identity/chat binding model.

14.3. Есть Telegram inbox/notification model для критичных событий и summaries.

14.4. Есть useful read surfaces для publish / readiness / jobs / ops / problem contexts.

14.5. Есть mutating operator actions через Telegram для ключевых remote workflows.

14.6. Все mutating actions auditable, permission-checked, idempotent и guarded.

14.7. Telegram layer не ломает канонические доменные semantics и не создает second source of truth.

14.8. Telegram layer usable до Epic 6 redesign.

15. Предлагаемая декомпозиция на микрофичи

ветка: feature/e6a-mf1-telegram-identity-gateway

ветка: feature/e6a-mf2-telegram-notifications-inbox

E6A-MF3 — Telegram Publish Approvals & Manual Handoff Controls
ветка: feature/e6a-mf3-telegram-publish-approvals

E6A-MF4 — Telegram Status, Readiness & Queue Views
ветка: feature/e6a-mf4-telegram-status-readiness

E6A-MF5 — Telegram Ops, Recovery & Safe Batch Controls
ветка: feature/e6a-mf5-telegram-ops-batch-controls

E6A-MF6 — Telegram Audit, Safety, Idempotency & Operator Hardening
ветка: feature/e6a-mf6-telegram-audit-safety

16. Что готово

16.1. Зафиксирован Epic 6A как отдельный epic-level operator domain.
16.2. Зафиксирована роль Telegram как remote operator console.
16.3. Зафиксированы scope boundaries, action model, safety model и decomposition на микрофичи.

17. Чего не хватает

17.1. Для REQ уровня эпика критических пробелов не осталось.
17.2. Следующий шаг — переход к SPEC или к первой микрофиче:
E6A-MF1 — Telegram Identity, Chat Binding & Action Gateway Foundation

18. Риски двусмысленности

18.1. Нельзя quietly превратить Epic 6A в “AI chat agent” вместо structured operator console.
18.2. Нельзя quietly расширить Telegram layer до uncontrolled admin surface.
18.3. Нельзя смешивать Telegram remote surface и canonical domain ownership.
18.4. Нельзя silently перенести Epic 6 redesign scope в Epic 6A.

READY FOR SPEC: YES


REQ_E6A_MF1_TELEGRAM_IDENTITY_CHAT_BINDING_AND_ACTION_GATEWAY_FOUNDATION_v1.0

1. Название

E6A-MF1 — Telegram Identity, Chat Binding & Action Gateway Foundation
ветка: feature/e6a-mf1-telegram-identity-gateway

2. Цель

Создать foundation для Epic 6A, которая:
- связывает Telegram identities и product operator identities;
- определяет trusted chat/thread contexts;
- вводит permission-aware action gateway;
- обеспечивает канонический базовый transport/control contract для всех следующих Telegram microfeatures.

3. Зачем делаем

Без MF1 Telegram layer останется набором ad-hoc bot handlers без канонической модели:
- кто имеет право действовать;
- в каком chat/thread context;
- какие действия разрешены;
- как защищать callbacks/actions от stale/idempotency проблем.

MF1 нужен, чтобы все последующие Telegram actions и surfaces строились на одном security/control foundation.

4. Scope v1

В MF1 входит:

4.1. Telegram operator identity model
- mapping Telegram user -> product operator identity
- active/inactive operator status for Telegram access
- permission scope linkage

4.2. Chat binding model
- trusted private chat binding
- trusted group/thread binding where allowed
- revocation/disable model
- binding state visibility

4.3. Action gateway foundation
- canonical action envelope
- callback/command routing contract
- target context resolution
- permission precheck contract
- stale-state precheck contract hook
- idempotency hook foundation

4.4. Access classes
Минимальные permission classes в v1:
- read-only
- standard operator mutate
- guarded operator mutate
- privileged operator mutate

4.5. Operator bootstrap surface
- enrollment / registration flow
- identity check / whoami / health-style diagnostic surface
- misbinding/error visibility

5. Non-goals v1

В MF1 не входит:
- rich domain read views;
- notifications/inbox lifecycle;
- publish approvals;
- ops/recovery actions;
- batch actions;
- deep audit hardening details beyond required foundation.

6. Ключевые product-решения

6.1. Product operator identity — source of truth.
Telegram identity только привязывается к ней.

6.2. Telegram chat/thread не может сам по себе выдавать права.
Права определяются только через product operator identity + explicit binding.

6.3. Action gateway должен быть общим и reusable.
Нельзя плодить domain-specific ad-hoc callback semantics по каждому будущему Telegram flow.

6.4. Permission model должна быть coarse-grained enough для Epic 6A v1, но с явным различением read-only / mutate / privileged.

6.5. Любой mutating flow downstream обязан идти через gateway foundation MF1.

7. Operator model

7.1. Один оператор может иметь ограниченный набор допустимых Telegram bindings.
7.2. Один chat/thread context может быть разрешен или запрещен для конкретного action class.
7.3. Revoked/invalid binding must fail closed.
7.4. Operator identity mismatch must not silently proceed.

8. Gateway contract requirements

8.1. Любое Telegram action intent downstream должно передавать:
- operator identity context
- chat/binding context
- target entity reference
- action type
- action freshness/expiry context
- correlation/audit context

8.2. Gateway обязан предоставить:
- permission check result
- stale target precheck hook
- idempotency key/correlation hook
- user-readable error classification

9. Acceptance criteria

MF1 считается успешно закрытым на уровне REQ, если:

9.1. Есть каноническая модель Telegram identity -> product operator identity.
9.2. Есть каноническая модель trusted chat/thread bindings.
9.3. Есть reusable action gateway foundation.
9.4. Есть permission classes для downstream flows.
9.5. Любой future Telegram mutating flow должен естественно опираться на MF1, а не обходить его.

10. Зависимости

Это foundation-микрофича для всего Epic 6A.
От нее зависят:
- MF2
- MF3
- MF4
- MF5
- MF6

11. Что готово

11.1. Зафиксирована каноническая remote-operator роль Telegram.
11.2. Зафиксировано, что Telegram не должен становиться second source of truth.

12. Чего не хватает

12.1. Для REQ уровня MF1 критических пробелов не осталось.

13. Риски двусмысленности

13.1. Нельзя смешивать identity model и chat model.
13.2. Нельзя делать ad-hoc permission checks вместо общей gateway foundation.
13.3. Нельзя допускать mutating flows вне canonical action gateway.

READY FOR SPEC: YES


REQ_E6A_MF2_TELEGRAM_NOTIFICATIONS_INBOX_AND_MESSAGE_LIFECYCLE_v1.0

1. Название

E6A-MF2 — Telegram Notifications, Inbox & Message Lifecycle
ветка: feature/e6a-mf2-telegram-notifications-inbox

2. Цель

Добавить в Epic 6A полноценную Telegram notification/inbox layer, чтобы оператор получал:
- critical alerts;
- actionable event messages;
- summaries/digests;
- pending/problem follow-ups;
- понятный message lifecycle вместо разрозненных одноразовых уведомлений.

3. Зачем делаем

Remote operator console без controlled notification/inbox model быстро превращается либо в шумный спам-канал, либо в unreliable alert surface.
MF2 нужен, чтобы Telegram стал usable operator inbox:
- с понятной категоризацией сообщений;
- с severity;
- с follow-up логикой;
- с closed/resolved behavior;
- с actionable continuity.

4. Scope v1

В MF2 входит:

4.1. Message families
- critical alerts
- actionable alerts
- summaries/digests
- unresolved follow-ups
- resolution/closure updates where meaningful

4.2. Message classification
Каждое сообщение должно иметь:
- category
- severity
- target context
- mutability/actionability class
- stale/replaced/resolved behavior

4.3. Inbox lifecycle
- active
- superseded
- resolved
- expired
- non-actionable informational

4.4. Delivery rules
- routing to correct bound operator/chat context
- dedupe expectations
- quiet/noisy class distinction
- summary vs immediate alert behavior

4.5. Operator acknowledgment semantics
- acknowledge message
- open related context
- escalate to deeper flow
- keep unresolved if action not taken

5. Non-goals v1

В MF2 не входит:
- full domain action surfaces beyond light acknowledgment/open;
- publish-specific action semantics in full detail;
- ops/recovery mutation flows;
- analytics-specific notification domain;
- general chat history/search product.

6. Ключевые product-решения

6.1. Telegram inbox должен быть stateful по message lifecycle.
Нельзя считать каждое сообщение одноразовым fire-and-forget.

6.2. Не каждое событие должно идти как immediate critical alert.
Нужна категоризация:
- critical now
- actionable normal
- digest-worthy
- follow-up only

6.3. Message lifecycle должен уметь отражать устаревание.
Если context изменился, старое сообщение не должно выглядеть как still-current action source.

6.4. Inbox layer должен уменьшать operator confusion, а не увеличивать его.

7. Обязательные event families v1

- publish/manual handoff events
- readiness/blocker events
- recovery/ops alerts
- worker/factory health alerts
- stale/problem follow-up notifications
- digest summaries

8. Acceptance criteria

8.1. Есть каноническая notification/inbox taxonomy.
8.2. Есть message lifecycle states.
8.3. Есть routing/delivery rules.
8.4. Есть acknowledgment/follow-up model.
8.5. Старые/stale сообщения не притворяются still-current operator source.

9. Зависимости

Зависит от:
- MF1

Используется далее в:
- MF3
- MF4
- MF5
- MF6

10. Что готово

10.1. В проекте уже есть Telegram notifications around publish flows как базовый вектор.
10.2. Epic 6A поднимает это до canonical inbox layer.

11. Чего не хватает

11.1. Для REQ уровня MF2 критических пробелов не осталось.

12. Риски двусмысленности

12.1. Нельзя quietly превратить inbox в spam channel.
12.2. Нельзя потерять message lifecycle statefulness.
12.3. Нельзя смешивать acknowledgment и real domain mutation.

REQ_E6A_MF3_TELEGRAM_PUBLISH_APPROVALS_AND_MANUAL_HANDOFF_CONTROLS_v1.0

1. Название

E6A-MF3 — Telegram Publish Approvals & Manual Handoff Controls
ветка: feature/e6a-mf3-telegram-publish-approvals

2. Цель

Сделать Telegram канонической remote surface для publish-domain operator flows, чтобы оператор мог удаленно:
- просматривать publish decision contexts;
- approve/reject where allowed;
- работать с manual handoff;
- подтверждать/manual-complete related flows;
- выполнять publish-related next actions безопасно и auditable.

3. Зачем делаем

Publish lifecycle — один из самых критичных удаленных operator сценариев.
В проекте уже есть Telegram action flows вокруг publish, но Epic 6A должен превратить их в полноценную продуктовую surface, а не частный набор bot handlers.

4. Scope v1

В MF3 входит:

4.1. Publish decision surfaces
- approval/reject decision contexts
- manual handoff pending contexts
- state/problem explanation in Telegram form
- transition-safe action presentation

4.2. Manual handoff remote support
- acknowledge manual handoff
- confirm manual completion where allowed
- carry minimal necessary context
- support operator follow-through from Telegram

4.3. Publish action safeguards
- stale-state protection
- permission-aware action exposure
- explicit confirmation for risky transitions
- state-changed handling

4.4. Context compression
Telegram message должен давать компактно:
- target release/job context
- current publish state
- reason/blocker
- available next actions
- link to web UI when full detail is needed

5. Non-goals v1

В MF3 не входит:
- redesign publish domain itself;
- full publish executor logic;
- analytics/prediction decisions;
- general queue browsing across all domains;
- ops/recovery actions outside publish-adjacent flows.

6. Ключевые product-решения

6.1. MF3 не invent’ит новые publish semantics.
Он лишь дает Telegram surface для уже канонических publish-domain actions.

6.2. Publish actions из Telegram должны быть state-safe.
Если underlying publish state изменился, действие не должно silently succeed on stale assumptions.

6.3. Manual handoff — first-class Telegram scenario, а не fallback afterthought.

6.4. Telegram publish controls должны покрывать реальную удаленную работу, а не быть “approve button only”.

7. Обязательные Telegram publish flows v1

- approve
- reject
- acknowledge manual handoff
- mark manual completion where policy/domain allows
- related publish-status/problem inspection
- operator-readable conflict/error outcomes

8. Acceptance criteria

8.1. Есть полноценные Telegram publish approval/reject flows.
8.2. Есть Telegram manual handoff support.
8.3. Все действия auditable, permission-checked and stale-safe.
8.4. Telegram publish controls не создают second publish logic outside canonical domain model.

9. Зависимости

Зависит от:
- MF1
- MF2

Связан с:
- Epic 3 domain contracts

10. Что готово

10.1. Базовый publish-oriented Telegram direction already exists in project context.
10.2. Epic 6A formalizes and expands it into a remote control surface.

11. Чего не хватает

11.1. Для REQ уровня MF3 критических пробелов не осталось.

12. Риски двусмысленности

12.1. Нельзя quietly добавить новые publish transitions через Telegram.
12.2. Нельзя свести MF3 к “уведомления плюс одна кнопка”.
12.3. Нельзя терять manual handoff как полноценный remote workflow.

READY FOR SPEC: YES


REQ_E6A_MF4_TELEGRAM_STATUS_READINESS_AND_QUEUE_VIEWS_v1.0

1. Название

E6A-MF4 — Telegram Status, Readiness & Queue Views
ветка: feature/e6a-mf4-telegram-status-readiness

2. Цель

Добавить в Telegram компактные read-first operator surfaces, чтобы удаленно можно было быстро понять:
- общее состояние фабрики;
- что сломано или заблокировано;
- какие publish/manual items требуют внимания;
- какие release/job contexts проблемные;
- какие очереди/каналы требуют оператора.

3. Зачем делаем

Remote operator console не может состоять только из alerts и approve-buttons.
Оператору нужен быстрый read surface для понимания:
- что сейчас происходит;
- где проблема;
- насколько это срочно;
- что открывать дальше.

4. Scope v1

В MF4 входит:

4.1. Factory overview summaries
- high-level status
- critical counts
- problem concentrations
- operator attention summary

4.2. Queue views
- publish queue summary
- manual handoff queue
- blocked/problem queue
- retry/attention-needed queue where relevant

4.3. Readiness views
- readiness blockers
- pre-publish readiness summaries

REQ_E6A_MF5_TELEGRAM_OPS_RECOVERY_AND_SAFE_BATCH_CONTROLS_v1.0

1. Название

E6A-MF5 — Telegram Ops, Recovery & Safe Batch Controls
ветка: feature/e6a-mf5-telegram-ops-batch-controls

2. Цель

Добавить в Telegram ограниченный, guardrailed набор remote ops/recovery/batch controls, чтобы оператор мог удаленно:

3. Зачем делаем

- ограниченными;

6.3. Dangerous actions должны либо иметь stronger guardrails, либо оставаться web-only outside v1.

6.4. Telegram remote ops не должен silently diverge from web/admin behavior.

7. Обязательные action classes v1

- safe retry/ack style actions
- safe recovery actions
- safe bounded batch actions

8. Acceptance criteria

8.3. Все такие действия проходят confirmation where required.

- MF1

REQ_E6A_MF6_TELEGRAM_AUDIT_SAFETY_IDEMPOTENCY_AND_OPERATOR_HARDENING_v1.0

1. Название

E6A-MF6 — Telegram Audit, Safety, Idempotency & Operator Hardening
ветка: feature/e6a-mf6-telegram-audit-safety

2. Цель

Добавить cross-cutting hardening layer для всего Epic 6A, чтобы Telegram operator console была:
- auditable;
- idempotent;
- stale-safe;
- permission-safe;
- explainable in failures;
- production-like in operator trustworthiness.

3. Зачем делаем

Без MF6 Telegram surface останется полезной, но потенциально опасной:
- callback на старом сообщении;
- повторный клик вызывает дубль;
- действие прошло на уже изменившемся state;
- оператор не понимает, почему действие отклонено;
- аудит Telegram-triggered changes недостаточен.

MF6 нужен, чтобы Epic 6A был production-grade, а не просто удобным ботом.

4. Scope v1

В MF6 входит:

4.1. Audit model
- audit trail for all Telegram-triggered actions
- correlation to operator identity/chat/action/target/result
- distinguish read access vs mutating access where meaningful

4.2. Idempotency model
- repeated callbacks/commands
- duplicate delivery/click behavior
- safe re-entry rules
- deterministic user-visible outcomes

4.3. Stale/conflict handling
- expired messages
- outdated target state
- already completed actions
- cross-surface conflict behavior

4.4. Safety messaging
User-visible error/result classes for:
- permission denied
- expired action
- stale state
- entity missing
- action already applied
- unsafe in current state
- system unavailable / degraded

4.5. Operator hardening rules
- callback expiry
- fail-closed semantics
- restricted action windows where needed
- strong confirmation for risky actions
- safe degradation behavior

5. Non-goals v1

В MF6 не входит:
- redesign of domain audit systems outside Telegram context;
- general enterprise IAM product;
- new domain semantics for publish/ops/planner;
- generic chat moderation features.

6. Ключевые product-решения

6.1. Telegram action without audit is not acceptable.
Auditability обязательна.

6.2. Telegram action without idempotency is not acceptable.
Repeated user interaction must not create unsafe duplicates.

6.3. Telegram action on stale state must fail safely and clearly.

6.4. User-visible failures must be operator-readable.
Не технический мусор, а practical operator result.

6.5. If there is uncertainty, Telegram layer fails closed.

7. Acceptance criteria

7.1. Все Telegram mutating actions auditable.
7.2. Все Telegram mutating actions idempotent.
7.3. Expired/stale/cross-surface conflict cases handled safely.
7.4. Operator receives clear action result/error semantics.
7.5. Epic 6A becomes production-grade and trustworthy.

8. Зависимости

Зависит от:
- MF1
- MF2
- MF3
- MF4
- MF5

9. Что готово

9.1. На уровне Epic 6A уже зафиксированы требования к guarded, permission-checked, stale-safe actions.
9.2. MF6 собирает это в explicit hardening microfeature.

10. Чего не хватает

10.1. Для REQ уровня MF6 критических пробелов не осталось.

11. Риски двусмысленности

11.1. Нельзя рассматривать hardening как “необязательный polish”.
11.2. Нельзя допускать mutating Telegram actions без strong audit/idempotency guarantees.
11.3. Нельзя fail open при stale/uncertain conditions.

READY FOR SPEC: YES
