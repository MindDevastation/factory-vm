REQ_PROMPT_REGISTRY_REAL_EXECUTION_RUNTIME_v1.0

1. Current state / baseline

Prompt Registry dispatch foundation is closed after MF32 and is intentionally read-only.
Existing closed foundation includes:
- dispatch attempt ledger
- review/detail/recheck/readiness surfaces
- execution guard placeholder
- capability matrix
- audit preview
- preflight summary
- operator checklist
- operator handoff
- handoff export/download
- handoff preview UI
- MF31 contract snapshots freezing current read-only foundation
- MF32 closure handoff confirming real execution runtime is not implemented

Current baseline therefore means:
- dispatch can prepare, review, export, and hand off an execution intent
- dispatch cannot safely execute linked actions
- there is no real runtime lifecycle, no real runtime state ownership, and no real execution-side operator control

2. Problem statement

The current Prompt Registry foundation stops at review and handoff. It does not provide:
- controlled execution of whitelisted linked actions
- explicit operator-confirmed dispatch execution
- runtime state ownership and status visibility
- idempotent and duplicate-safe execution admission
- crash-safe async execution and recovery
- auditable execution writes
- usage and lifecycle event writes for real runtime activity
- safe retry and cancellation handling

Without a real execution runtime, Prompt Registry remains a review/export subsystem rather than an operational execution layer.

3. Goals

P0
- turn dispatch execution from placeholder/read-only foundation into a controlled runtime
- support only explicitly whitelisted linked actions
- require explicit operator confirmation before execution
- make execution idempotent and duplicate-safe
- write immutable audit trail for all execution attempts and outcomes
- expose execution status clearly in API and UI
- remain safe if worker crashes mid-run
- prevent uncontrolled external execution
- prevent secret leakage
- preserve Basic Auth on operator/admin surfaces
- remain SQLite-compatible

P1
- support retries where allowed
- support cancellation where allowed
- integrate with queue/job/worker runtime safely
- support append-only lifecycle event timeline and usage history
- support operator-visible stale/conflict outcomes

P2
- support operator ergonomics for recheck, retry, cancel, and result inspection
- support later extensibility for new linked action families without redesigning the runtime core

All included items in scope are mandatory.

4. Non-goals

- no production code in this REQ
- no branch creation
- no APPLY
- no uncontrolled external execution
- no direct execution from preview/detail/readiness pages
- no free-form autonomous agent behavior
- no replacement of Basic Auth with a new auth model
- no general-purpose workflow engine for the whole product
- no real YouTube upload tests
- no network dependencies in tests
- no hidden domain semantics beyond explicitly whitelisted linked action capabilities
- no secret material exposed to operators, exports, audit preview, or UI surfaces

5. Actors / operator roles

Operator roles for v1:
- Read-only operator
  - can inspect review/detail/recheck/readiness
  - can inspect audit-safe execution details, statuses, and timelines
  - cannot confirm, retry, cancel, or force execution

- Standard operator
  - can confirm execution for non-privileged whitelisted actions
  - can inspect execution detail and result
  - can retry retryable executions for non-privileged actions
  - can cancel execution before irreversible commit where cancellation is allowed

- Privileged operator
  - can confirm privileged whitelisted actions
  - can retry privileged retryable actions
  - can cancel privileged executions where cancellation is allowed
  - can perform runtime recovery actions such as reclaim/release/cancel-on-conflict where explicitly permitted

- Runtime worker/system
  - claims admitted async executions
  - updates runtime states
  - renews lease while running
  - writes terminal result
  - never bypasses safety gates or capability matrix

Canonical rule:
- product operator identity is the source of truth for permissions
- execution rights are determined by product operator identity plus action capability class
- Basic Auth remains required for all admin/operator runtime surfaces

6. Supported action types for v1

Whitelisted linked action capabilities for v1 are exact and closed to the following list:
- CREATE_BULK_JSON_DRAFT
- CREATE_METADATA_REQUEST
- CREATE_VISUAL_REQUEST
- CREATE_ANALYTICS_REQUEST
- ENQUEUE_INTERNAL_PROMPT_JOB
- GENERATE_OPERATOR_HANDOFF_EXPORT

Capability semantics:
- CREATE_BULK_JSON_DRAFT
  - creates or updates an internal bulk JSON draft target from a confirmed rendered prompt payload
- CREATE_METADATA_REQUEST
  - creates an internal metadata request target using a confirmed rendered prompt payload
- CREATE_VISUAL_REQUEST
  - creates an internal visual workflow request target using a confirmed rendered prompt payload
- CREATE_ANALYTICS_REQUEST
  - creates an internal analytics/recommendation workflow request target using a confirmed rendered prompt payload
- ENQUEUE_INTERNAL_PROMPT_JOB
  - creates and enqueues an internal runtime job for approved prompt-linked processing
- GENERATE_OPERATOR_HANDOFF_EXPORT
  - generates a controlled operator handoff/export artifact from the confirmed rendered prompt payload

Hard runtime restrictions:
- no direct arbitrary shell execution
- no arbitrary network calls
- no external AI provider execution from runtime v1
- no unregistered capability execution
- no capability execution outside capability matrix allow rules

7. Execution lifecycle

Canonical lifecycle for a single execution attempt:
- PREPARED
- PREFLIGHT_REJECTED or CONFIRMATION_REQUIRED or STALE_BLOCKED or CONFLICT_BLOCKED
- CONFIRMATION_REQUIRED
- ADMITTED
- DISPATCHED or RUNNING
- RUNNING
- SUCCEEDED or RETRY_PENDING or FAILED_TERMINAL or CANCELLED
- RETRY_PENDING
- ADMITTED again on explicit retry
- terminal outcome

Canonical lifecycle phases:
1. Reviewable execution intent exists in read-only foundation.
2. Runtime preflight checks current capability, permissions, freshness, target state, secret-safety, and dedup state.
3. Runtime returns one of:
   - PREFLIGHT_REJECTED
   - STALE_BLOCKED
   - CONFLICT_BLOCKED
   - CONFIRMATION_REQUIRED
4. Operator performs explicit confirmation.
5. Runtime admits exactly one execution attempt and writes initial audit/lifecycle records.
6. Runtime dispatches execution to sync adapter or async queue path.
7. Runtime runs action and updates state.
8. Runtime writes terminal result, usage summary, and audit trail.

Canonical execution-mode mapping:
- synchronous only:
  - CREATE_BULK_JSON_DRAFT
  - CREATE_METADATA_REQUEST
  - CREATE_VISUAL_REQUEST
  - CREATE_ANALYTICS_REQUEST
- asynchronous only:
  - ENQUEUE_INTERNAL_PROMPT_JOB
  - GENERATE_OPERATOR_HANDOFF_EXPORT

There is no capability in v1 that may choose sync or async dynamically.

9. Safety gates

Every execution attempt must pass the following gates in this exact order:
1. capability exists in capability matrix
2. capability is execution-enabled
3. operator identity is authenticated under Basic Auth
4. operator has permission for capability class
5. prompt version is executable and not archived/inactive for runtime use
6. binding resolution is complete and unambiguous
7. rendered payload validation succeeds
8. target entity exists
9. target entity state is compatible with capability
10. reviewed target snapshot hash matches current target snapshot hash
11. dedup/idempotency gate passes
12. concurrency lock can be acquired
13. secret-safe adapter precheck passes
14. operator explicitly confirms the exact execution intent

Hard safety rules:
- no execution from page load
- no execution from preview-only action
- no silent execution from detail/review/recheck/readiness pages
- no execution when target snapshot changed after review without new confirm cycle
- no execution when capability matrix says review-only
- no execution when secret-safe precheck fails

10. Idempotency and deduplication

Execution runtime is fully idempotent.

Canonical dedup key for v1:
sha256(
  capability_code |
  target_type |
  target_id |
  prompt_record_id |
  prompt_version_id |
  binding_resolution_fingerprint |
  rendered_payload_hash |
  action_payload_hash |
  reviewed_target_state_hash
)

Rules:
- duplicate confirm clicks with the same dedup key must resolve to the same admitted execution record
- retries do not reuse the original execution record; they create a new execution attempt linked to the same execution group and same dedup lineage
- a new review/recheck that changes rendered payload, binding resolution, or target snapshot produces a new dedup key
- operator identity is not part of dedup key; duplicate execution must collapse across operators if the execution intent is the same

11. Locks / concurrency

Canonical concurrency model for v1:
- one execution group lock per (capability_code, target_type, target_id)
- one execution record lease for async running attempts
- one stale confirmation guard bound to reviewed_target_state_hash

Required lock behavior:
- target capability lock is acquired before ADMITTED
- async worker lease is acquired before RUNNING
- stale locks are reclaimable after lease expiry
- only one RUNNING or DISPATCHED attempt may exist per execution group at a time
- RETRY_PENDING does not release execution-group conflict semantics for duplicate admission until resolved by retry/cancel/terminal close

Crash safety:
- if worker dies mid-run, lease eventually expires
- reclaimed execution must resume or terminalize the same execution attempt without creating duplicate side effects
- crash recovery never bypasses audit/event writes

13. Usage/event writes

Three distinct write streams are mandatory.

Audit writes:
- compliance/security/operator trace
- one immutable event per guarded runtime milestone

Lifecycle event writes:
- append-only state progression stream
- exact transition timeline for UI/detail/recheck/readiness
- written on every state transition including retry scheduling and reclaim

Usage writes:
- one usage record is created at ADMITTED
- the same usage record is updated with terminal outcome summary
- usage record stores:
  - execution_group_id
  - first_admitted_attempt_id
  - latest_attempt_id
  - prompt_record_id
  - prompt_version_id
  - rendered_payload_hash
  - binding_resolution_fingerprint
  - capability_code
  - target_type
  - target_id
  - operator_id who admitted execution
  - first_admitted_at
  - terminal_outcome
  - terminal_at
  - artifact_ref if any

Usage writes are not created for PREFLIGHT_REJECTED, STALE_BLOCKED, or CONFLICT_BLOCKED attempts.
Those are audit and lifecycle events only.

15. Retry policy

Retry policy is exact for v1.

Retry eligibility:
- only asynchronous capabilities may enter RETRY_PENDING
- synchronous capabilities never auto-retry
- synchronous failures are terminal unless operator explicitly starts a new execution attempt from a new confirm cycle

Retry-eligible capabilities:
- ENQUEUE_INTERNAL_PROMPT_JOB
- GENERATE_OPERATOR_HANDOFF_EXPORT

Retry counts and backoff:
- max automatic retries per execution attempt: 3
- retry schedule: exponential backoff 30s, 120s, 600s
- after retries are exhausted: FAILED_TERMINAL with result code RETRIES_EXHAUSTED

Retry invariants:
- retry preserves dedup lineage and execution_group_id
- retry creates a new attempt_id
- retry requires the same capability/target lineage and same admitted payload lineage
- retry must be skipped and converted to STALE_BLOCKED or CONFLICT_BLOCKED if target or binding state is no longer compatible

16. Rollback / cancellation boundaries

Exact cancellation boundaries for v1:

CREATE_BULK_JSON_DRAFT
- cancellation allowed before commit
- after commit no automatic rollback
- follow-up compensating action is separate domain action outside this runtime

CREATE_METADATA_REQUEST
- cancellation allowed before commit
- after commit no automatic rollback

CREATE_VISUAL_REQUEST
- cancellation allowed before commit
- after commit no automatic rollback

CREATE_ANALYTICS_REQUEST
- cancellation allowed before commit
- after commit no automatic rollback

ENQUEUE_INTERNAL_PROMPT_JOB
- cancellation allowed until downstream queue admission is committed
- after downstream queue admission there is no generic rollback
- later domain-specific cancel/reset is separate from runtime rollback

GENERATE_OPERATOR_HANDOFF_EXPORT
- cancellation allowed until artifact finalize commit
- partial/incomplete artifact must be discarded on cancellation
- finalized artifact is not automatically rolled back; later delete/expire is separate lifecycle

Global rollback rule:
- runtime v1 does not provide generic automatic rollback after irreversible commit
- compensation is explicit and action-family specific, not hidden runtime magic

17. Queue/job/worker integration

Canonical model for v1:
- hybrid execution runtime
- synchronous capabilities execute inside request-cycle adapter boundary
- asynchronous capabilities execute through a dedicated logical execution queue built on the existing worker fabric

Exact async ownership rules:
- async execution records are admitted in runtime store first
- dedicated logical queue item is created for runtime execution
- worker claims queue item by lease
- worker updates execution attempt state and heartbeats lease as needed
- on success/failure it writes terminal outcome and releases lease
- on crash/timeout a reclaim path may resume same attempt or terminalize it safely

There is no direct fire-and-forget path.
Every async execution must have a persisted execution record before queue admission.
