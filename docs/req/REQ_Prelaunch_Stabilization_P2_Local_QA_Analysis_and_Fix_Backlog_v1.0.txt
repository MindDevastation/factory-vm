REQ_PRELAUNCH_STABILIZATION_P2_LOCAL_QA_ANALYSIS_AND_FIX_BACKLOG_v1.0

1. Название

Pre-Launch Stabilization — Punkt 2
Local Structured QA Analysis and Additional Fix Backlog

2. Цель

Локально поднять проект, провести структурированный ручной QA и по его итогам собрать формализованный backlog дополнительных фиксов перед запуском на VPS.

3. Зачем делаем

Цель пункта 2 — не "покликать проект" и не собрать wishlist, а:
- подтвердить, что проект реально поднимается локально;
- проверить критические операторские сценарии;
- выявить P0 / P1 / P2 проблемы до VPS запуска;
- сформировать структурированное ТЗ на дополнительные фиксы.

4. Канонический подход

4.1. Пункт 2 фиксируется как structured local pre-launch QA pass.

4.2. Во время QA фиксы не вносятся.
Сначала:
- выполняется полный проход;
- findings фиксируются;
- проводится приоритизация.
Только потом формируется отдельный fix backlog.

4.3. Findings нельзя смешивать с новыми фичами.
По итогам QA отдельно существуют:
- реальные defects;
- UX / operability improvements;
- new feature requests вне текущего pre-launch scope.

5. Scope

В scope пункта 2 входит:

5.1. Startup & environment sanity
- локальный запуск проекта;
- старт сервисов;
- env/config sanity;
- базовый smoke по UI/API.

5.2. Core operator flows
- Planner;
- release / materialization / job creation;
- queue / render path;
- publish-related path;
- operator critical flows.

5.3. Recovery / stale / edge behavior
- повторные действия;
- invalid inputs;
- partial failures;
- stale action handling;
- operator-safe recovery behavior.

5.4. Reports / exports
- отчеты;
- export surfaces;
- поведение при отсутствии данных / ошибках.

5.5. Telegram local QA
- optional-if-available;
- не является launch blocker, если локальная настройка мешает основному QA.

6. Non-goals

В пункт 2 не входит:
- redesign;
- визуальная полировка ради красоты;
- нагрузочное тестирование;
- бесконечный exploratory QA;
- автоматическое включение всех minor improvements в pre-launch fix pack;
- новые фичи, не являющиеся findings локального QA.

7. QA Levels

7.1. Level A — Startup & viability
Проверяется:
- проект поднимается;
- критичных стартовых проблем нет.

Status:
- works.

7.2. Level B — Core end-to-end flows
Проверяются:
- Planner;
- release/job creation;
- queue/render;
- publish path;
- operator critical flows.

7.3. Level C — Recovery / stale / edge behavior
Проверяются:
- invalid input handling;
- stale errors;
- preview/execute separation;
- recovery after partial failure;
- correct no-create behavior when validation fails.

7.4. Level D — Operability
Проверяются:
- понятность действий;
- интуитивность flows;
- operator friction;
- перегруз страниц;
- discoverability primary actions.

8. Severity model

8.1. P0 — launch blocker
- проект не стартует;
- критический сценарий не проходит;
- опасное поведение;
- severe corruption risk.

8.2. P1 — production-significant defect
- важный операторский сценарий сломан;
- высокая вероятность operator error;
- плохая recoverability;
- важный flow ненадежен.

8.3. P2 — important usability / workflow friction
- не блокирует запуск;
- но заметно мешает стабильной эксплуатации.

8.4. P3 — polish / nice-to-have
- не входит автоматически в предзапусковый пакет фиксов.

9. Findings format

Для каждого finding фиксируется:
- ID
- severity
- where found
- preconditions
- exact reproduction steps
- expected behavior
- actual behavior
- impact
- suspected scope
- proposed fix direction
- must-fix-before-VPS: yes/no

14. Output of Punkt 2

Результатом пункта 2 является:
- structured QA findings set;
- must-fix-before-VPS backlog;
- post-launch improvement backlog;
- база для отдельного fix implementation pass.

15. Definition of done

Пункт 2 считается завершенным, если:
- Level A подтвержден;
- Levels B/C/D пройдены достаточно для pre-launch freeze;
- findings разделены на defects / UX improvements / new features;
- сформирован must-fix-before-VPS список;
- сформирован postpone-after-launch список.

16. Что готово

16.1. Structured local QA pass проведен.
16.2. Найден набор critical defects и UX frictions.
16.3. New feature requests отделены от QA backlog.
16.4. Есть clear must-fix-before-VPS list.

17. Чего не хватает

17.1. Для REQ уровня пункта 2 критических пробелов не осталось.
17.2. Следующий шаг — отдельный fix implementation pass по must-fix backlog.

18. Риски двусмысленности

18.1. Нельзя смешивать QA findings и новые фичи.
18.2. Нельзя автоматически тянуть P2 UX items в launch-blocking scope.
18.3. Нельзя считать Planner umbrella finding закрытым без полного follow-up прохода после фиксов.
