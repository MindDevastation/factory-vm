SPEC_GROWTH_INTELLIGENCE_ONE_CLICK_RELEASE_PLANNING_v1.1

1. Overview / цель

Growth Intelligence + One-Click Release Planning defines a product feature that:
- uses real YouTube metrics and channel history
- interprets them through knowledge base / playbooks / trust model
- generates a release plan for week / month / quarter in one click
- works in operator-approved planning mode in v1
- may later evolve toward automatic planning under stricter policy and safety rules

The purpose of the feature is not to merely display analytics, but to convert analytics + heuristics + trust into actionable, explainable release planning.

This v1.1 spec preserves the base feature and adds stronger foundation-layer requirements for:
- trustworthy knowledge handling
- reproducible explainability
- rollout safety
- typed integrations
- future ML readiness
- structured operator feedback

3. Canonical product model

3.1. Feature identity
The feature must be implemented as:
Growth Intelligence + One-Click Release Planning

3.2. Four-layer architecture
The feature must contain four mandatory layers:
- Knowledge Base
- Trust / Impact Model
- Planning Engine
- Operator Surfaces

3.3. Primary product goal
The feature must:
- analyze signals
- activate playbooks
- build release plans
- explain why the plan exists
- show trust
- provide safe handoff/application paths
- remain operator-approved by default

4. Primary scenarios

4.1. Scenario A — One-click planning
Operator selects:
- channel
- planning period: week / month / quarter
- mode:
  - safe
  - balanced
  - growth
  - experiment-heavy

Operator presses Generate Plan.

System returns:
- release schedule by dates
- recommended release types / formats
- goals
- suggested publish windows
- suggested packaging focus
- risks
- trust category
- expected impact
- linked recommendations
- linked actions

4.2. Scenario B — Rebuild plan after new data
After metrics refresh, the system:
- recalculates the plan
- shows what changed
- explains why it changed
- shows what should be replaced / moved / strengthened

4.3. Scenario C — Assisted planning
Operator may request:
- best slots next week
- top releases with highest growth potential
- releases that should be postponed

6. Signal model

6.1. Analytical signals
The system must use at minimum:
- CTR
- retention
- watch time
- avg view duration
- views / impressions
- traffic sources
- audience segments
- unique viewers
- search terms
- geographies
- publish window behavior
- trend / anomaly state
- historical release performance
- comparisons by period

6.2. Derived signals
The system must support at minimum:
- packaging problem
- timing problem
- format fatigue
- topic momentum
- search opportunity
- browse weakness
- audience mismatch
- overposting risk
- underposting risk
- safe-format candidate
- high-upside experiment candidate

6.3. Knowledge / playbook signals
The system must support:
- official YouTube guidance
- practitioner heuristics
- internal proven heuristics
- channel-specific learned heuristics

7. Trust model

7.1. Trust model structure
The trust model must use two independent scales.

7.2. Source Trust scale
Frozen Source Trust categories:
- A — Official / strongly validated
- B — strong practitioner pattern
- C — conditional / niche / format-specific
- D — experimental hypothesis

7.3. Impact Confidence scale
Frozen Impact Confidence categories:
- High
- Medium
- Low

7.4. Final displayed trust
Operator-facing trust must combine the two scales, for example:
- Trust: A / High
- Trust: B / Medium
- Trust: C / High but niche

7.5. Trust display rule
Every meaningful plan recommendation or slot suggestion must expose trust in operator-readable form.

8. Knowledge registry model

8.1. Knowledge item baseline fields
Knowledge registry must support content-bearing fields for:
- title
- description
- source_type / source_name
- trigger_conditions
- action_template
- explanation_template
- applicable_profiles
- applicable_metrics
- trust fields
- status

8.2. Provenance contract for knowledge items
Every knowledge item must also store provenance fields:
- source_url
- source_class
- evidence_note
- reviewed_by
- reviewed_at
- supersedes_item_id
- invalidated_at

8.3. Frozen source_class literals
Knowledge provenance must use:
- OFFICIAL
- PRACTITIONER
- INTERNAL

8.4. Provenance rule
Trust must be verifiable, not declarative only.
The system must distinguish official, practitioner, and internal knowledge sources explicitly.

8.5. Review lifecycle rule
Knowledge items must support review lifecycle semantics through:
- reviewed_by
- reviewed_at
- supersedes_item_id
- invalidated_at

9. Research bootstrap model

9.1. Seed bootstrap requirement
The system must support initial population of the knowledge registry from a curated research pack.

9.2. Bootstrap ordering rules
- official high-signal sources are imported first
- practitioner sources are imported as second layer
- low-trust or weakly supported sources must not automatically become trusted defaults
- import must be repeatable
- import must be audit-friendly

9.3. Bootstrap purpose rule
Bootstrap must avoid an empty knowledge registry at start and shorten time to first useful planning output while preserving source hierarchy.

10. Planning engine model

10.1. Required inputs
Planning engine must use:
- current analyzer state
- historical analyzer state
- profile (channel strategy + format profile)
- active signals / risks / opportunities
- rules / playbooks / trust matrix
- planning horizon
- operator mode (safe / balanced / growth / experiment-heavy)

10.2. Required outputs
Planning engine must produce:
- ranked release slots
- recommended release mix
- safe actions
- growth bets
- experiment slots
- packaging / timing guidance
- risk warnings
- approval-ready plan

10.3. Mandatory constraints
Planning engine must:
- not violate approved publishing cadence
- not overload risky slots
- not schedule too many experiments in a row
- not auto-apply without permission
- preserve explainability for each decision

11. Experiment budget policy

11.1. Explicit policy requirement
The feature must define an explicit experiment-budget policy rather than relying only on vague heuristics.

11.2. Minimum policy fields
Experiment budget policy must support:
- max_experiments_per_week
- max_experiments_per_month
- max_high_risk_slots_per_horizon
- channel_specific_risk_budget
- cooldown_after_failed_experiment

11.3. Planning constraint rule
The planning engine and lint layer must enforce experiment-budget policy when evaluating plan validity.

12. Immutable explanation snapshot

12.1. Snapshot requirement
Every planning run must persist an immutable explanation snapshot.

12.2. Minimum snapshot contents
The immutable explanation snapshot must include:
- normalized input signals snapshot
- resolved trust mapping
- activated playbooks
- reasoning snapshot
- linked action resolution snapshot
- version markers for relevant planning policy

12.3. Reproducibility rule
The immutable explanation snapshot exists to support:
- reproducibility
- plan comparison
- post-hoc explanation of why the system suggested a specific plan

13. Typed linked-actions catalog

13.1. Linked-actions typing rule
Linked actions must not remain free text.
They must use a typed action taxonomy.

13.2. Minimum typed actions
The system must support at minimum:
- OPEN_ANALYZER
- OPEN_PLANNER
- OPEN_METADATA_SURFACE
- SEND_TELEGRAM_DIGEST
- EXPORT_PLAN

13.3. Extension rule
Additional actions are allowed only through explicit action taxonomy extension.

13.4. Integration rule
Typed linked actions must improve:
- UI wiring
- predictability of integrations
- clarity of action resolution

14. Typed planner handoff contract

14.1. Typed handoff rule
Planner handoff must use typed payload instead of weakly structured free content.

14.2. Minimum planner handoff payload
Planner handoff payload must support:
- slot_intent
- release_goal
- packaging_direction
- metadata_emphasis
- fallback_path
- source_reasoning_reference

14.3. Stability rule
Planner handoff must be stable, testable, and ambiguity-resistant between planning surfaces and planner surfaces.

15. Plan lint / validation layer

15.1. Mandatory lint pass
Before a generated plan is shown in UI or Telegram, the system must run a required validation/lint pass.

15.2. Minimum lint checks
Lint must check at minimum:
- missing trust on meaningful recommendation
- missing reasoning
- missing source signals
- empty or unresolved linked actions
- conflicting slot recommendations
- too many risky/experiment slots in a row
- empty required fields
- experiment budget violations

15.3. Invalid-plan rule
If lint fails:
- the plan must not be treated as valid
- the operator must see structured invalid reasons
- invalid state must be captured in audit trail
