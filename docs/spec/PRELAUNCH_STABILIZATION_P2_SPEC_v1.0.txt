SPEC_PRELAUNCH_STABILIZATION_P2_LOCAL_QA_ANALYSIS_AND_FIX_BACKLOG_v1.0

1. Overview / цель

Pre-Launch Stabilization — Punkt 2 defines a structured local pre-launch QA program and fix-backlog preparation pass for factory-vm.

Its purpose is to:
- confirm that the project actually starts locally;
- verify critical operator flows;
- identify P0 / P1 / P2 issues before VPS launch;
- produce a structured fix backlog for the next implementation pass.

This is not exploratory clicking, not redesign discovery, and not a feature wishlist exercise.

2. Canonical operating model

2.1. Punkt 2 is a structured local pre-launch QA pass.

2.2. During QA, fixes are not applied.
The required order is:
- full structured QA pass
- finding capture
- prioritization
- explicit backlog formation

2.3. Findings must be separated into:
- real defects
- UX / operability improvements
- new feature requests outside current pre-launch scope

2.4. QA findings must not be mixed with future feature work.

3. Scope / Non-goals

3.1. In scope
- local startup and service bring-up
- env/config sanity
- basic UI/API smoke
- core operator flows:
  - Planner
  - release / materialization / job creation
  - queue / render path
  - publish-related path
  - operator-critical flows
- recovery / stale / edge behavior:
  - repeated actions
  - invalid inputs
  - partial failures
  - stale action handling
  - operator-safe recovery behavior
- reports / exports:
  - report surfaces
  - export surfaces
  - empty-data and error behavior
- Telegram local QA only if locally available

3.2. Out of scope
- redesign
- visual polish for beauty
- load testing
- endless exploratory QA
- automatic inclusion of all minor improvements into pre-launch fix pack
- new features that are not findings from the local QA pass

3.3. Telegram rule
Telegram local QA is optional-if-available and is not a launch blocker if local setup interferes with core QA.

4. QA level model

4.1. Level A — Startup & viability
Purpose:
- confirm project starts locally
- confirm no critical startup failures

4.2. Level B — Core end-to-end flows
Required scope:
- Planner
- release/job creation
- queue/render
- publish path
- operator critical flows

4.3. Level C — Recovery / stale / edge behavior
Required scope:
- invalid input handling
- stale errors
- preview/execute separation
- recovery after partial failure
- correct no-create behavior when validation fails

4.4. Level D — Operability
Required scope:
- clarity of actions
- intuitiveness of flows
- operator friction
- page overload
- discoverability of primary actions

5. Severity model

5.1. P0 — launch blocker
Definition:
- project does not start
- critical scenario does not pass
- dangerous behavior
- severe corruption risk

5.2. P1 — production-significant defect
Definition:
- important operator scenario is broken
- high probability of operator error
- poor recoverability
- important flow is unreliable

5.3. P2 — important usability / workflow friction
Definition:
- does not block launch
- but materially harms stable operation

5.4. P3 — polish / nice-to-have
Definition:
- does not automatically enter pre-launch fix pack

5.5. PASS
Definition:
- checked area passed the current QA scope

6. Finding format contract

Every finding must include:
- ID
- severity
- where found
- preconditions
- exact reproduction steps
- expected behavior
- actual behavior
- impact
- suspected scope
- proposed fix direction
- must-fix-before-VPS: yes/no

12. Definition of done

Punkt 2 is done only when all of the following are true:
- Level A is confirmed
- Levels B/C/D are covered sufficiently for pre-launch freeze
- findings are split into defects / UX improvements / new features
- must-fix-before-VPS list is formed
- postpone-after-launch list is formed

13. Validation rules

13.1. QA process validation
- fixes must not be applied during Punkt 2
- findings capture must happen before prioritization
- prioritization must happen before fix-pass planning

13.2. Finding classification validation
- real defects must not be mixed with new features
- P2 UX items must not be auto-promoted into launch-blocking scope
- umbrella findings must not be treated as closed until explicit follow-up coverage is complete

13.3. Backlog validation
- must-fix backlog must contain only pre-launch required items
- post-launch backlog must contain only explicitly non-blocking items
- excluded new-feature requests must remain outside Punkt 2 fix scope

14. Rollback / containment principles

14.1. QA containment rule
Punkt 2 produces findings and backlog only. It does not mutate product behavior.

14.2. Scope containment rule
If an observation is a redesign request, exploratory wish, or new feature, it must be kept out of the must-fix QA backlog unless separately reclassified by product decision.

14.3. Follow-up containment rule
Planner umbrella finding B-PL-03 is not considered closed until a specific post-fix follow-up pass confirms closure.

15. Open questions / ambiguities

15.1. Exact document/artifact formatting for findings export is implementation detail as long as the frozen finding structure, must-fix list, and postponed list remain explicit.

15.2. Exact local Telegram QA depth is implementation detail because Telegram is optional-if-available and not a launch blocker if local setup interferes with core QA.
