SPEC_PROMPT_RUNTIME_AUTHORITATIVE_GATE_FOUNDATION_v1.0

1. Overview / цель

Prompt Runtime Authoritative Gate Foundation defines the prerequisite authority layer that must exist before Prompt Registry real execution runtime can be honestly finalized.

Its purpose is to provide persisted, auditable, fail-closed authoritative sources for:
- capability enablement
- operator permission class
- rendered payload validity
- canonical target resolver identity
- canonical target compatibility policy
- canonical target snapshot and snapshot hash
- composite runtime gate evaluation

This feature is not the execution runtime itself.
It is the authority layer that execution runtime must read before:
- admitting execution
- confirming execution eligibility
- resolving runtime targets
- trusting rendered payloads

2. Scope / Non-goals

2.1. In scope
- persisted capability enablement registry keyed by capability_code
- persisted operator-to-runtime-permission-class mapping keyed by authenticated Basic Auth subject
- persisted render validation ledger keyed by:
  - prompt version
  - binding fingerprint
  - render result hash
- persisted canonical target resolver registry keyed by:
  - capability_code
  - target_type
- persisted target compatibility policy keyed by:
  - capability_code
  - target_type
- persisted target snapshot ledger with canonical payload and deterministic snapshot hash
- service contracts that execution runtime can call for gate evaluation
- admin/operator read APIs
- admin/operator maintenance APIs where the REQ allows them
- audit-safe inspection surfaces
- fail-closed behavior whenever authoritative source is absent, stale, invalid, disabled, deprecated, or unresolved

2.2. Out of scope
- production execution runtime implementation
- queue/worker redesign
- UI redesign
- arbitrary target-type expansion beyond explicitly registered supported target types
- hidden permission system outside the explicit model
- shell/provider/network execution adapters
- YouTube-specific behavior
- execution attempt writes beyond what is required for authoritative gates themselves
- preview/detail/review page execution
- synthetic defaults when authoritative data is absent
- migration away from Basic Auth
- non-SQLite requirements

2.3. Boundary rules
- this foundation itself does not execute actions
- this foundation must not synthesize gate truth from UI state
- this foundation must not synthesize permission truth from ad hoc code assumptions
- this foundation must not synthesize compatibility truth locally in runtime code
- all authoritative sources must be persisted and auditable
- any missing source must cause fail-closed blocked evaluation

3. Problem statement

3.1. Baseline problem
Current Prompt Registry dispatch foundation is intentionally read-only and does not provide authoritative persisted gate sources for real runtime admission.

3.2. Confirmed missing authoritative sources
- capability execution-enabled source
- operator permission / capability-class source
- rendered payload validity source
- canonical runtime target resolver / snapshot / compatibility contract

3.3. Consequence rule
Without these sources, any “real execution runtime” would have to rely on:
- inferred local code paths
- hardcoded assumptions
- stale preview/review data
- ad hoc target resolution
- synthetic permission truth

That is forbidden for spec-honest runtime admission.

4. Canonical architecture role

4.1. Feature role
This feature is the authoritative gate foundation for later execution runtime.

4.2. Downstream dependency rule
Execution runtime must consume this feature as the only trusted gate source for:
- capability enablement
- operator permission
- render validity
- target resolver identity
- target compatibility
- target snapshot truth

4.3. Fail-closed rule
If any authoritative gate source is missing, invalid, disabled, deprecated, unresolved, or stale beyond contract, downstream runtime must return blocked outcome rather than continue.

5. Canonical data model

5.1. prompt_runtime_capability_registry
Purpose:
authoritative source of whether a capability is execution-enabled and what runtime permission class is required.

Required columns:
- id
- capability_code
- execution_enabled
- required_permission_class
- status
- notes nullable
- updated_by_operator
- created_at
- updated_at

Required uniqueness:
- unique(capability_code)

Frozen enum: required_permission_class
- runtime_view
- runtime_execute
- runtime_operate
- runtime_admin

Frozen enum: status
- active
- disabled
- deprecated

Runtime admissibility rule:
A capability is execution-admissible only when:
- row exists
- execution_enabled = true
- status = active

5.2. prompt_runtime_operator_permissions
Purpose:
authoritative mapping from authenticated operator identity to runtime permission class.

Canonical operator subject source:
- Basic Auth username

Required columns:
- id
- operator_subject
- permission_class
- is_enabled
- notes nullable
- updated_by_operator
- created_at
- updated_at

Required uniqueness:
- unique(operator_subject)

Frozen enum: permission_class
- runtime_view
- runtime_execute
- runtime_operate
- runtime_admin

Permission order:
- runtime_admin > runtime_operate > runtime_execute > runtime_view

Runtime rule:
If operator row is absent or disabled, gate evaluation must fail closed.

5.3. prompt_runtime_render_validation_ledger
Purpose:
authoritative proof that a rendered payload was validated for runtime use.

Required columns:
- id
- prompt_record_id
- prompt_version_id
- binding_fingerprint
- render_result_hash
- validation_status
- validation_schema_version
- validator_code
- validated_at
- invalid_reason_code nullable
- invalid_reason_detail nullable
- superseded_by_validation_id nullable
- created_at

Frozen enum: validation_status
- passed
- failed
- error
- superseded

Canonical latest-validation uniqueness boundary:
- prompt_version_id
- binding_fingerprint
- render_result_hash

Runtime rule:
Execution runtime may trust rendered payload only if latest authoritative row for the exact tuple is:
- present
- validation_status = passed
- not superseded

5.4. prompt_runtime_target_resolver_registry
Purpose:
authoritative registry of which resolver is canonical for a given capability and target type.

Required columns:
- id
- capability_code
- target_type
- resolver_code
- snapshot_schema_version
- is_enabled
- notes nullable
- updated_by_operator
- created_at
- updated_at

Required uniqueness:
- unique(capability_code, target_type)

Runtime rule:
If no enabled resolver exists for exact capability_code + target_type, runtime must fail closed.

5.5. prompt_runtime_target_compatibility_policy
Purpose:
authoritative policy for whether a capability may run against a target type.

Required columns:
- id
- capability_code
- target_type
- compatibility_status
- policy_code
- notes nullable
- updated_by_operator
- created_at
- updated_at

Required uniqueness:
- unique(capability_code, target_type)

Frozen enum: compatibility_status
- allowed
- blocked
- deprecated

Runtime rule:
Execution runtime may proceed only when exact row exists and compatibility_status = allowed.

5.6. prompt_runtime_target_snapshot_ledger
Purpose:
authoritative persisted target snapshot used by execution runtime for stale/conflict-safe admission.

Required columns:
- id
- capability_code
- target_type
- target_ref
- resolver_code
- snapshot_schema_version
- snapshot_payload_json
- snapshot_hash
- compatibility_status_at_capture
- resolved_at
- created_at

Canonical snapshot envelope fields
The outer snapshot envelope is frozen and must always include:
- target_type
- target_ref
- target_display_label
- target_state_code
- target_exists
- target_updated_at or equivalent state freshness marker
- compatibility_inputs
- resolver_metadata

Resolver-owned inner payload rule:
Per-target inner payload is resolver-owned but must remain compatible with the outer envelope contract.

Snapshot hash contract:
- snapshot_hash must be computed over canonical serialized snapshot payload
- field ordering and serialization rules must be deterministic
- same target state must produce same hash under same schema version

Runtime rule:
If target snapshot cannot be resolved, cannot be hashed deterministically, or is incompatible, runtime must fail closed.

6. Required indexes

6.1. P0 required indexes
- unique index on prompt_runtime_capability_registry.capability_code
- unique index on prompt_runtime_operator_permissions.operator_subject
- composite index on prompt_runtime_render_validation_ledger(prompt_version_id, binding_fingerprint, render_result_hash, validated_at)
- unique index on prompt_runtime_target_resolver_registry(capability_code, target_type)
- unique index on prompt_runtime_target_compatibility_policy(capability_code, target_type)
- composite index on prompt_runtime_target_snapshot_ledger(capability_code, target_type, target_ref, resolved_at)

7. Internal service contracts

7.1. CapabilityGateService
Purpose:
resolve authoritative capability gate by capability_code

Input:
- capability_code

Output:
- capability_code
- execution_enabled
- required_permission_class
- status
- authoritative_source_metadata

Fail-closed behavior:
- no row => reject
- disabled/deprecated => reject

7.2. OperatorPermissionService
Purpose:
resolve authoritative permission class for authenticated operator

Input:
- operator_subject

Output:
- operator_subject
- permission_class
- is_enabled

Fail-closed behavior:
- no row => reject
- disabled => reject

7.3. RenderValidationService
Purpose:
resolve latest authoritative validation state for rendered payload tuple

Input:
- prompt_version_id
- binding_fingerprint
- render_result_hash

Output:
- latest_validation_record
- authoritative_verdict:
  - trusted
  - untrusted
  - missing

Fail-closed behavior:
- missing row => reject
- latest row failed/error/superseded => reject

7.4. TargetResolverService
Purpose:
resolve canonical target snapshot for exact capability_code + target_type + target_ref

Input:
- capability_code
- target_type
- target_ref

Output:
- canonical_target_snapshot_payload
- snapshot_hash
- resolver_metadata
- compatibility_inputs

Fail-closed behavior:
- no enabled resolver => reject
- target missing/unresolvable => reject
- non-deterministic or incomplete snapshot => reject

7.5. TargetCompatibilityService
Purpose:
check authoritative compatibility between capability and target type

Input:
- capability_code
- target_type

Output:
- compatibility_status
- policy_code
- policy_metadata

Fail-closed behavior:
- no compatibility row => reject
- blocked/deprecated => reject

7.6. PromptRuntimeGateEvaluationService
Purpose:
compose all authoritative sources into one runtime admission evaluation

Input:
- operator_subject
- capability_code
- prompt_version_id
- binding_fingerprint
- render_result_hash
- target_type
- target_ref

Output:
- admission_status
- required_permission_class
- resolved_permission_class
- render_validation_status
- target_compatibility_status
- target_snapshot_hash
- failure_reason_code
- authoritative_source_summary

Allowed admission_status:
- admissible
- blocked_missing_authority
- blocked_disabled_capability
- blocked_permission
- blocked_invalid_render
- blocked_target_resolution
- blocked_target_compatibility

Composite fail-closed rule:
Any missing/disabled/invalid/unresolved authoritative source must return a blocked outcome.

9. Runtime invariants

9.1. P0 invariants
1. No execution runtime may trust a capability unless capability registry row exists and is active/enabled.
2. No execution runtime may trust operator authorization unless operator permission row exists and is enabled.
3. No execution runtime may trust rendered payload unless latest exact validation ledger row is passed.
4. No execution runtime may trust target eligibility unless:
   - enabled resolver exists for exact capability_code + target_type
   - compatibility policy exists for exact capability_code + target_type
   - target snapshot resolves successfully
   - snapshot hash is deterministic
5. Any missing authoritative source causes fail-closed block.
6. No preview/detail/recheck page may silently substitute inferred truth when authoritative source is absent.
7. No gate source may be synthesized from UI state only.
8. Basic Auth username is canonical operator subject until later auth redesign explicitly replaces it.
9. Target compatibility is determined only by authoritative compatibility policy row, not by ad hoc runtime guesswork.
10. Snapshot freshness is determined by resolved canonical snapshot and hash, not by stale preview payload.
11. Compatibility and resolver truth must be inspectable and auditable.
12. This foundation itself does not execute actions.

9.2. P1 invariants
13. Latest authoritative render validation for an exact tuple supersedes older rows.
14. Operator permission hierarchy must be monotonic and comparable.
15. Deprecated capability or compatibility rows do not silently coerce to allowed.
16. Snapshot schema version must be explicit and persisted.

10. Status / versioning support

10.1. P1 status requirements
Versioning/status support must exist where needed for registries and policies.

10.2. Explicit state/status rule
Enabled/disabled/deprecated/invalid states must be persisted explicitly where REQ requires them.
